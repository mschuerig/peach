## CRITICAL/HIGH ISSUES

### H1: AC 7 — Minimum-selection guard test is missing

**AC 7** explicitly requires: *"minimum-selection guard is tested."* No such test exists in `SettingsTests.swift` or anywhere else. The guard is only enforced at the UI layer (`IntervalSelectorView.swift:47,59`) but never verified by a unit test.

The minimum-selection logic (`isLastActive = isActive && selection.intervals.count == 1`) is a behavioral invariant that should be testable independently of the SwiftUI view. Since the project convention is "views are thin" and doesn't require SwiftUI view testing, this logic could be extracted into a testable helper or tested via `IntervalSelection` itself.

**Location:** Missing test in `PeachTests/Settings/SettingsTests.swift`
**Fix:** Add a test verifying that when only one interval remains selected, the guard prevents removal. Could test at the `IntervalSelection` or toggle-logic level.

---

## MEDIUM ISSUES

### M1: No domain-level validation rejects empty intervals set

`IntervalSelection` happily accepts and round-trips an empty set (`[]`). The UI prevents reaching this state, but if UserDefaults is manually set to `"[]"`, the app would operate with zero selected intervals. `AppUserSettings.intervals` would return an empty `Set<DirectedInterval>`, which downstream consumers (training sessions) may not handle gracefully.

**Location:** `Peach/Settings/IntervalSelection.swift:10-16`
**Fix:** Either reject empty sets in `init?(rawValue:)` (return nil, triggering default fallback) or add a validation in `AppUserSettings.intervals` to fall back when the decoded set is empty.

### M2: Tests use shared `UserDefaults.standard` without isolation

Tests in `SettingsTests.swift:42-65` write to and read from `UserDefaults.standard`, which is a shared singleton. Swift Testing can run tests concurrently within a `@Suite`. If multiple interval-related tests run simultaneously, they could interfere with each other. While the tests clean up after themselves, a test failure before cleanup would leak state.

**Location:** `PeachTests/Settings/SettingsTests.swift:42-65`
**Fix:** Consider using a custom `UserDefaults(suiteName:)` for test isolation, or mark these tests as serialized (though this is a pre-existing pattern in the test file).

---

## LOW ISSUES

### L1: Non-deterministic serialization order

`Set<DirectedInterval>` has no guaranteed iteration order. `JSONEncoder` encodes the set as a JSON array with potentially varying element order across runs. While functionally harmless (decode produces the same set regardless), sorting before encoding would make stored values deterministic and diff-friendly. `DirectedInterval` already conforms to `Comparable`.

**Location:** `Peach/Settings/IntervalSelection.swift:18-24`

### L2: Down-P1 cell mirrors active state of prime

When prime is selected, the Down-P1 cell shows accent color at 0.3 opacity (active but disabled). Since "down prime" doesn't exist conceptually (`.down(.prime)` normalizes to `.prime`), showing it as always inactive would be semantically clearer. The current visual treatment is acceptable but slightly misleading.

**Location:** `Peach/Settings/IntervalSelectorView.swift:40-44`

### L3: Silent data loss on encoding failure path

`IntervalSelection.rawValue` returns `"[]"` if `JSONEncoder` fails, silently replacing the user's selection with an empty set. Falling back to `IntervalSelection.default.rawValue` would preserve at least one interval. This path is extremely unlikely to trigger.

**Location:** `Peach/Settings/IntervalSelection.swift:18-24`
