### YELLOW MEDIUM ISSUES

**2. [MEDIUM] No tests for downward interval challenge generation in PitchMatchingSession**
`PeachTests/PitchMatching/PitchMatchingSessionTests.swift`

Same issue as #1. `PitchMatchingSession.generateChallenge()` at `PitchMatchingSession.swift:191-205` has identical direction-aware note range logic, tested only with `.prime` and `.up(.perfectFifth)`. The `.down` path is untested at the session integration level.

**3. [MEDIUM] `DirectedInterval.down(.prime)` creates semantically invalid but distinct value**
`Peach/Core/Audio/DirectedInterval.swift:20-22`

The `down()` factory doesn't validate that `interval != .prime`, allowing:
```swift
DirectedInterval.down(.prime).displayName == DirectedInterval.prime.displayName  // true — both "Prime"
DirectedInterval.down(.prime) == DirectedInterval.prime  // FALSE! Different values
Set([.prime, .down(.prime)]).count  // 2, not 1
```
"Prime down" is semantically meaningless (0 semitones in any direction = same note). This could cause subtle bugs if someone accidentally constructs `.down(.prime)` — it looks identical in UI but fails equality/Set checks.

**4. [MEDIUM] Stale docstring in Interval.swift**
`Peach/Core/Audio/Interval.swift:3`

The doc comment says *"Used by `MIDINote.transposed(by:)` to transpose a note"* — but this extension was removed from `Interval.swift` and moved to `DirectedInterval.swift`. Since this file was modified in this story, updating the docstring is in scope.

---

### GREEN LOW ISSUES

**5. [LOW] Inconsistent DirectedInterval factory usage in StartScreen**
`Peach/Start/StartScreen.swift:30,38` vs `:49,57`

Lines 30 and 38 use the verbose form `[DirectedInterval.prime]`, while lines 49 and 57 use the short form `[.up(.perfectFifth)]`. Since the parameter type is `Set<DirectedInterval>`, the compiler can infer `.prime` — should use `[.prime]` for consistency.

**6. [LOW] Redundant `Comparable` operator in Direction**
`Peach/Core/Audio/Direction.swift:14-16`

The explicit `<` operator comparing `rawValue` is redundant. Swift auto-synthesizes `Comparable` for enums from case declaration order (which matches rawValue order here: `up=0` before `down=1`). The explicit implementation works but is unnecessary code.

**7. [LOW] Missing downward transposition boundary test**
`PeachTests/Core/Audio/DirectedIntervalTests.swift`

While `transposeDown` tests the happy path (`MIDINote(67).transposed(by: .down(.perfectFifth))`), there's no edge case test for transposing down near MIDI 0 where the precondition would trigger. A boundary test would strengthen coverage.
