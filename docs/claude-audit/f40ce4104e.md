### CRITICAL/HIGH ISSUES

**1. [HIGH] Audio resource leak in `commitPitch()` from `awaitingSliderTouch` (orphaned handle)**

`PitchMatchingSession.swift:114-123` — When `commitPitch()` is called directly from `awaitingSliderTouch`, it calls `startTunableNote()` which spawns an async Task to play the tunable note. Then `commitResult()` runs synchronously *before* the async Task assigns the handle to `currentHandle`. In `commitResult()`, `handleToStop = currentHandle` captures `nil`, so the tunable note handle is never stopped.

The orphaned Task eventually completes, checks `state != .idle` (true — state is `.showingFeedback` or later), and assigns the handle to `currentHandle`. This handle's note continues playing through the feedback phase, the next reference note, and into the next cycle. It's only cleaned up on `stop()` (via `notePlayer.stopAll()`), not in the normal training loop.

While in typical DragGesture usage `adjustPitch` fires before `commitPitch` (so the `awaitingSliderTouch` check in `commitPitch` is bypassed), this code path is explicitly implemented and tested (`commitPitchFromAwaitingSliderTouchProducesResult`), making it a real production risk.

**Fix:** In `commitPitch()`, when transitioning from `awaitingSliderTouch`, don't call `startTunableNote()`. The tunable note will immediately be stopped by `commitResult()` anyway — starting it just to stop it is wasteful and creates the race. Instead, set `pendingTunableFrequency = nil` and proceed directly to `commitResult()`.

---

### MEDIUM ISSUES

**2. [MEDIUM] Unstructured Task in `startTunableNote()` is not cancellable**

`PitchMatchingSession.swift:189-211` — The `Task { ... }` spawned by `startTunableNote()` is unstructured with no stored reference. When `stop()` is called, it cancels `trainingTask` and `feedbackTask` but cannot cancel this task. The guard `state != .idle` provides protection, but a rapid `stop()` → `start()` sequence could allow the old Task to assign its handle into the new session (since state is no longer `.idle` after `start()`).

**Fix:** Store the Task reference and cancel it in `stop()`, or use structured concurrency by making `startTunableNote()` async and calling it within the existing task hierarchy.

**3. [MEDIUM] First `adjustPitch` frequency adjustment is silently dropped**

`PitchMatchingSession.swift:101-112` — When `adjustPitch()` transitions from `awaitingSliderTouch`, it calls `startTunableNote()` (async) then immediately tries `currentHandle?.adjustFrequency(...)`. Since `currentHandle` is still `nil` (the async play hasn't completed), the optional chain is a no-op. The user's initial drag position is effectively ignored.

The dev notes document this as "acceptable because the initial cent offset is already applied to the tunable frequency," but this means the first few drag events during rapid interaction produce no audible response. Subsequent events work correctly once the handle is assigned.
