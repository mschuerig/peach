### CRITICAL/HIGH ISSUES

**1. HIGH: PerceptualProfileTests `pitchMatchingObserverUsesReferenceNoteWithInterval` doesn't actually verify referenceNote keying**
`PerceptualProfileTests.swift:234-251`

The test claims "PitchMatchingObserver uses referenceNote as profile key when interval is non-prime", but the assertions only check global accumulators:
```swift
#expect(profile.matchingSampleCount == 1)
#expect(profile.matchingMean != nil)
```

The pitch matching profile's `updateMatching(note:centError:)` method (`PerceptualProfile.swift:119-126`) doesn't index by note at all тАФ it updates global `matchingCount`/`matchingMeanAbs`/`matchingM2`. These assertions pass regardless of whether `referenceNote` or `targetNote` is used. The test is trivially true and verifies nothing about note keying.

Compare with the comparison observer test (`PerceptualProfileTests.swift:209-232`), which correctly verifies `statsForNote(MIDINote(60)).sampleCount == 1` and `statsForNote(MIDINote(67)).sampleCount == 0` тАФ a real, falsifiable assertion.

**Fix:** Either make the test actually verify something meaningful (e.g., pass a wrong note and confirm different behavior) or rewrite the test description to match what it actually validates (that the observer records a result at all). Given the matching profile is a global aggregate, the best fix is renaming the test to accurately describe what it validates.

### MEDIUM ISSUES

**2. MEDIUM: IntervalTests displayName tests are tautological**
`IntervalTests.swift:153-166`

```swift
#expect(Interval.prime.displayName == String(localized: "Prime"))
#expect(Interval.perfectFifth.displayName == String(localized: "Perfect Fifth Up"))
```

Since `displayName` returns `String(localized: "Prime")` and the test expects `String(localized: "Prime")`, this is comparing the same expression to itself. The test can never fail (unless `displayName` maps `.prime` to the wrong key). The debug log notes this was a deliberate locale-independence fix, but it means these tests don't verify actual content тАФ only that the enum case maps to itself.

**Fix:** These tests should verify the localization KEY is correctly mapped per case. Since there's no API to extract the key from `String(localized:)`, consider at minimum testing with a `Bundle` approach or accepting this is a smoke test and updating the test descriptions to say "maps to correct localization key" rather than implying content verification.

**3. MEDIUM: PitchMatchingScreen interval label has no padding (inconsistent with ComparisonScreen)**
`PitchMatchingScreen.swift:10-37` vs `ComparisonScreen.swift:21-50`

`ComparisonScreen` has `.padding()` on its outer VStack (line 51), giving the interval label proper inset from screen edges. `PitchMatchingScreen`'s VStack has no `.padding()` тАФ only the `VerticalPitchSlider` has `.padding()` (line 27). The interval `Text` label sits in the VStack with no horizontal padding, so it could render edge-to-edge (though it centers naturally for short strings).

**Fix:** Add `.padding(.horizontal)` to the interval label in `PitchMatchingScreen`, or add `.padding()` to the outer VStack to match `ComparisonScreen`'s pattern.

**4. MEDIUM: Only 3 of 13 intervals have individual displayName tests тАФ 10 remain uncovered**
`IntervalTests.swift:151-179`

The story says "at minimum" test prime, perfectFifth, and octave. The remaining 10 intervals have no individual key verification. If `.minorThird` was accidentally mapped to `"Major Third Up"`, no test would catch it. The `uniqueDisplayNames` test would pass as long as all strings differ, but wouldn't detect a swap between two intervals.

**Fix:** Add a parameterized test covering all 13 intervals and their expected localization keys, or add a test verifying each case's key matches its expected string pattern.
