### MEDIUM ISSUES

**M1: VoiceOver accessibility regression — invisible indicator is readable**
**File:** `Peach/PitchMatching/PitchMatchingScreen.swift:21-26`

The old overlay approach used `if state == .showingFeedback { ... }`, so the indicator was only in the view tree when visible. The new always-render approach keeps the indicator in the tree at opacity 0. When `lastResult?.userCentError` is non-nil but state is not `.showingFeedback`, VoiceOver can still read the stale accessibility label (e.g., "4 cents sharp") from the invisible indicator. This is confusing for VoiceOver users.

**Fix:** Add `.accessibilityHidden(pitchMatchingSession.state != .showingFeedback)` to the indicator.

---

**M2: No tests for `PitchMatchingScreen.feedbackAnimation` static method**
**File:** Missing `PeachTests/PitchMatching/PitchMatchingScreenTests.swift`

The analogous `ComparisonScreen.feedbackAnimation` has two tests in `ComparisonScreenFeedbackTests.swift` (lines 126-133). `PitchMatchingScreen.feedbackAnimation` is untested. The project convention says "New layout logic → extract to `static` methods for unit testability" and the pattern exists in the codebase. This is a pre-existing gap but consistent with what the project expects.

---

### LOW ISSUES

**L1: Fixed 130pt frame thin on margin for extreme Dynamic Type**
**File:** `Peach/PitchMatching/PitchMatchingScreen.swift:24`

At standard type: icon (100pt max) + spacing (4pt) + `.title2` (~22pt) = ~126pt, leaving only 4pt margin in the 130pt frame. At accessibility text sizes (AX3+), `.title2` could grow to 30+pt, pushing content to ~134pt and clipping. Low probability in practice but worth awareness.
