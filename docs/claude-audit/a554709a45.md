## Adversarial Review: Dependency Analysis

Content type: **Full production codebase (57 Swift files), dependency direction and coupling**

---

### Findings

1. **Core/ has massive upward dependencies on feature modules.** `TrainingDataStore`, `PerceptualProfile`, `TrendAnalyzer`, `ThresholdTimeline`, `NextComparisonStrategy`, and `KazezNoteStrategy` — all living in `Core/` — depend on `Comparison`, `CompletedComparison`, and `ComparisonObserver` which are defined in `Comparison/`. Similarly, `TrainingDataStore` and `PerceptualProfile` depend on `CompletedPitchMatching` and `PitchMatchingObserver` from `PitchMatching/`. This is the dependency arrow pointing the wrong way: the foundational layer reaches upward into feature modules for types that are clearly shared domain concepts. These value types and observer protocols should live in `Core/`, not in the feature directories that happen to have introduced them first.

2. **Four Core/ files import `SwiftUI` for no domain reason.** `SoundFontLibrary.swift`, `TrendAnalyzer.swift`, `ThresholdTimeline.swift`, and `TrainingSession.swift` all `import SwiftUI` solely to declare `extension EnvironmentValues { @Entry var ... }`. The actual classes and protocols in these files have zero SwiftUI usage. This means your domain layer has a compile-time dependency on a UI framework because the environment key declarations are co-located with the domain logic instead of being extracted into a view-layer or app-layer file.

3. **`SettingsScreen` instantiates `TrainingDataStore` directly**, violating both the Dependency Inversion Principle and the project's own rule that "all service instantiation happens in `PeachApp.swift`." In `resetAllTrainingData()`, the view constructs `TrainingDataStore(modelContext: modelContext)` ad-hoc. The view should receive `TrainingDataStore` through `@Environment`, not build it from a raw `ModelContext`. This also means the view knows about `SwiftData`'s `ModelContext` — an implementation detail that should be hidden behind the data store abstraction.

4. **`ComparisonScreen` and `PitchMatchingScreen` have compile-time dependencies on concrete implementation types through `@Entry` default values.** The `@Entry` fallback closures reference `PerceptualProfile`, `KazezNoteStrategy`, `AppUserSettings`, and `MockHapticFeedbackManager`. Even though these defaults exist as fallbacks, they force the UI file to import and depend on concrete Core/ and Settings/ implementations. A screen file should not know that `KazezNoteStrategy` exists — that is an algorithm implementation detail.

5. **`SummaryStatisticsView` and `MatchingStatisticsView` use `PerceptualProfile` as a concrete type where protocols exist.** `computeStats(from:)` takes `PerceptualProfile` directly instead of `PitchDiscriminationProfile`. `computeMatchingStats(from:)` takes `PerceptualProfile` instead of `PitchMatchingProfile`. The protocols were designed for exactly this purpose — to let consumers depend on capability, not implementation — and they're being ignored.

6. **`AudioSessionInterruptionMonitor` (Core/) imports `UIKit`** for `UIApplication.didEnterBackgroundNotification` and `UIApplication.willEnterForegroundNotification`. A Core/ service should not depend on UIKit. These notification names could be injected or the monitor could accept notification names as parameters, or the notification constants could be referenced via their string equivalents without the UIKit import.

7. **`ComparisonSession` and `PitchMatchingSession` depend on `AudioSessionInterruptionMonitor` as a concrete type** with no protocol abstraction. This is a service that monitors system events — exactly the kind of dependency that should be behind a protocol for testability and to respect DIP. If you ever want a different interruption strategy or need to mock it in tests, you're stuck.

8. **`TrendAnalyzer` and `ThresholdTimeline` are injected into `ComparisonSession` as concrete types** with no protocol boundary. `ComparisonSession` stores `private let trendAnalyzer: TrendAnalyzer?` and `private let thresholdTimeline: ThresholdTimeline?` — concrete classes, not protocols. The session shouldn't need to know these are specifically `TrendAnalyzer` and `ThresholdTimeline`; they're just `ComparisonObserver` conformances that happen to also need resetting. If the reset behavior were part of a protocol, the session could be fully decoupled.

9. **Production code contains a mock.** `MockHapticFeedbackManager` lives in `Comparison/HapticFeedbackManager.swift` alongside the production `HapticFeedbackManager`. The project's own conventions state mocks belong in the test target. This mock pollutes the production binary and blurs the boundary between production and test code.

10. **Cross-feature coupling between `PitchMatching/` and `Comparison/`.** `PitchMatchingFeedbackIndicator` reaches into `ComparisonFeedbackIndicator.defaultIconSize` — a static property on a Comparison/ view — to share a UI constant. This creates a dependency from one independent training module to another. If the constant is shared, it should live in a shared location, or each module should define its own.

11. **`SoundFontNotePlayer` (Core/Audio/) depends on `SoundSourceID` from `Settings/`.** A Core/ implementation type reaches into the Settings/ module for a value type. `SoundSourceID` is a simple tag struct that arguably belongs in Core/ if Core/ needs it, or the dependency should flow through the `UserSettings` protocol without Core/ ever seeing the `SoundSourceID` type directly.

12. **`SettingsScreen` imports `SwiftData` and accesses `@Environment(\.modelContext)` directly.** The project rule says "only `TrainingDataStore` accesses SwiftData." By importing SwiftData and pulling the `ModelContext` from the environment, the settings view is coupled to the persistence framework's implementation. The data-reset operation should be a method on an injected service, not something the view assembles from raw framework primitives.
