## Architect's Technical Analysis

### Two Stop Paths Need Treatment

There are **two distinct stop paths** in the code, and both need the fade-out:

| Path | Location | Mechanism | When Used |
|------|----------|-----------|-----------|
| **Individual note stop** | `SoundFontPlaybackHandle.stop()` | `sampler.stopNote()` | Early answer during note 2, pitch matching stop |
| **All notes off** | `SoundFontNotePlayer.stopAll()` | MIDI CC#123 | Session cleanup, navigation away, backgrounding, audio interruption |

### Recommended Implementation

**The change point is `sampler.volume`** (inherited from `AVAudioNode`). This is separate from `sampler.overallGain` which controls MIDI amplitude — they don't interfere with each other.

**`SoundFontPlaybackHandle.stop()`** — currently at line 23:

```swift

    guard !hasStopped else { return }
    hasStopped = true
    sampler.stopNote(midiNote, onChannel: channel)
    sampler.sendPitchBend(SoundFontNotePlayer.pitchBendCenter, onChannel: channel)

// After:
func stop() async throws {
    guard !hasStopped else { return }
    hasStopped = true
    sampler.volume = 0
    try? await Task.sleep(for: SoundFontNotePlayer.stopPropagationDelay)
    sampler.stopNote(midiNote, onChannel: channel)
    sampler.sendPitchBend(SoundFontNotePlayer.pitchBendCenter, onChannel: channel)
    sampler.volume = 1.0
}
```

### Key Design Decisions

1. **`try?` on the sleep** — critical. Many callers use `try? await handle.stop()`. If the task is cancelled during the 25ms wait, `try?` ensures we still execute the actual stop and restore volume, rather than leaving the sampler muted.

2. **`sampler.volume` vs `engine.mainMixerNode.volume`** — `sampler.volume` is more targeted. The handle already holds a `sampler` reference, so no new dependencies needed.

3. **Constant shared via `SoundFontNotePlayer`** — the handle already references `SoundFontNotePlayer.pitchBendCenter`, so this follows the established pattern.

4. **No guard needed on `stopAll()`** — it's inherently idempotent (CC#123 on no playing notes is a no-op). The 25ms overhead on a redundant call is acceptable.

### Risk Assessment

- **Test impact:** Adds 25ms per stop call in integration tests. Minimal — tests already handle 20ms preset-load delays.
- **Behavioral impact on acoustic presets:** The 25ms mute will cut the very beginning of their natural release tails. At 25ms this is imperceptible — well below the ~100ms threshold for human temporal resolution of amplitude envelopes.
- **Concurrency:** Only one note plays at a time in both `ComparisonSession` and `PitchMatchingSession`, so no concurrent volume manipulation risk.
